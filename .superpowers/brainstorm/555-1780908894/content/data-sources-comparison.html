三种数据来源对比

Claude Code 监控工具的数据获取方式分析

方案 A：本地文件解析

数据最完整

优势

  • Token 数据：per-message 精度
  • 对话内容：完整 user/assistant 交互
  • 工具调用：含参数和结果
  • 文件变更：编辑历史备份
  • 项目维度：按项目/会话分组
  • Windows 兼容：完全支持

劣势

  • 非实时，需事后解析 JSONL
  • 需处理流式中间消息（token=0）
  • 文件可能较大，需增量解析

核心文件：.claude/projects/{slug}/{sessionId}.jsonl

方案 B：Anthropic API

数据最有限

优势

  • Console 有聚合用量统计
  • 可做整体预算监控
  • 实现简单

劣势

  • 无 per-session 数据
  • 无对话内容
  • 无工具调用详情
  • 需组织管理员权限
  • 与本地 JSONL 数据重复

仅适合整体预算监控，不适合精细监控

方案 C：Claude Code Hooks

实时性最强

优势

  • 实时事件推送：事件发生即捕获
  • PreToolUse / PostToolUse 捕获工具调用
  • UserPromptSubmit 捕获用户输入
  • Stop 捕获会话结束
  • 可推送到 HTTP 服务/数据库

劣势

  • 无 Token 数据（usage 仅在 assistant 消息中）
  • 无 assistant 响应内容
  • Windows 需 .cmd 包装器
  • timeout 较短（默认 10s）

可用事件：PreToolUse, PostToolUse, UserPromptSubmit, Stop, Notification

推荐：混合方案（A + C）

以本地 JSONL 文件解析为主数据源（获取完整 token、对话、工具调用数据），
以 Hooks 作为实时事件推送通道（实时感知工具调用和用户行为），
Anthropic Console API 仅用于整体预算监控（可选）。
