架构方案对比

两种整体架构方案，请选择你偏好的方向

A

一体化 Tauri 应用

Tauri 桌面应用同时承载悬浮窗 + Web 仪表盘。SpringBoot 后端仅做数据采集和存储。

架构： Tauri (前端) ←→ SpringBoot (后端 API)
悬浮窗： Tauri 多窗口，原生 always-on-top
仪表盘： Tauri 主窗口内嵌 WebView
数据采集： SpringBoot 后端定时解析 JSONL + Hooks 推送

优势

  • 单一应用，体验统一
  • Tauri 多窗口原生支持悬浮
  • 前端可复用（悬浮窗 + 仪表盘共享组件）

劣势

  • Tauri + SpringBoot 两套技术栈需同时维护
  • 动画资源需打包进应用
B

分离式：Tauri 悬浮窗 + 独立 Web 仪表盘

悬浮窗是独立的 Tauri 轻应用，Web 仪表盘由 SpringBoot 直接渲染。两者通过后端 API 共享数据。

架构： Tauri 悬浮窗 ←→ SpringBoot ←→ 浏览器仪表盘
悬浮窗： 独立 Tauri 应用，极简，只做实时状态
仪表盘： SpringBoot 内嵌前端，浏览器访问
数据采集： 同方案 A

优势

  • 悬浮窗可独立启动，更轻量
  • 仪表盘无需安装，浏览器即用
  • 两个模块解耦，可独立迭代

劣势

  • 两个入口，体验稍分散
  • 需分别启动服务

推荐方案 A：一体化 Tauri 应用

统一入口、统一体验。Tauri 的多窗口机制天然支持悬浮窗，主窗口即可承载仪表盘。
SpringBoot 后端专注数据采集和 API，Tauri 前端专注展示和交互。
